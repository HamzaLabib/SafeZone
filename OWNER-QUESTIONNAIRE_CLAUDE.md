---

## D4 — Form Submission Flow (Current Architecture)

1. User submits a form on the frontend.
2. The frontend sends a POST request to the appropriate API route (`/api/register-interest` or `/api/contact`).
3. The API route validates input, checks for honeypot spam field, enforces rate limiting (10 requests per 10 minutes per IP), and writes the submission to MongoDB.
4. If `RESEND_API_KEY`, `EMAIL_FROM`, and `ADMIN_NOTIFICATION_EMAIL` are all set, an email notification is dispatched via Resend to the admin.
5. The user is redirected to `/thank-you`.

No student confirmation email is currently sent — only an admin notification. If a student auto-reply is desired, it needs to be added to the API route.

---

## D5 — Admin Dashboard Access

- Route: `/admin/login` (not linked from any public navigation)
- Authentication: single shared password via HTTP-only cookie session (HMAC-SHA256 signed)
- Session duration: 8 hours
- Rate limiting is applied to the login endpoint
- **This is not production-grade multi-user authentication.** If multiple staff members need separate logins or access control, a proper auth system must be built before wider team rollout.
